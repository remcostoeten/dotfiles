Dotfiles Architecture

Development Environment Configuration
Personal development environment configuration and automation. Shell configurations, editor setups, custom utilities, and system preferences. Reproducible across machines with automated installation via OpenTUI-based setup tool.
Core Components
30+ custom scripts for daily workflows • Fish shell with extensive aliases and functions • Neovim configuration with LazyVim • Terminal setups (Kitty, WezTerm) • GNOME desktop customization • Automated installation system with retry logic and progress tracking

Directory Structure
Architecture

bin/

  • Executable wrappers on system PATH
  • Docker manager, wallpaper setter, VM controller
  • Script selector, alarm utility, emoji picker
  • License generator, clipboard tools

scripts/

  • Implementation logic for bin wrappers
  • TypeScript, Python, Fish, Bash implementations
  • Docker TUI application (499 lines)
  • Version manager, UI helpers, database tools

configs/

  • Fish shell (361 lines of aliases + functions)
  • Neovim LazyVim configuration
  • Kitty and WezTerm terminal configs
  • Git configuration with custom aliases
  • GNOME desktop settings (themes, extensions)

opentui-setup/

  • TypeScript installation system using OpenTUI
  • Interactive TUI and headless CLI modes
  • Package manager with 18 categories, 100+ packages
  • Retry logic, rollback, progress tracking
  • System verification and health checks

shell-setup/

  • Legacy shell-based installer (pre-OpenTUI)
  • Modular package groups and installers
  • Still functional, maintained for compatibility
  • Helper scripts for system configuration

env-private/

  • Machine-specific secrets and environment variables
  • API keys, tokens, private configuration
  • Gitignored to prevent accidental commits
  • Template files provided for reference
arch
Technical Architecture
Design
Tech Stack
Primary Languages: TypeScript, Bash, Fish Shell, Python
UI Framework: React with @opentui/react (TUI framework)
Runtime: Bun for TypeScript/JavaScript execution
Package Management: APT, Snap, NPM, GitHub Releases, CURL downloads
Shell: Fish (with extensive custom functions and aliases)
Editor: Neovim with LazyVim configuration
Directory Architecture
Separation of Concerns:
• `bin/` - Executables on PATH (wrappers, direct scripts)
• `scripts/` - Implementation logic (called by bin/)
• `configs/` - Application configurations (Fish, Neovim, Kitty, etc.)
• `opentui-setup/` - Interactive setup system (TypeScript/React)
• `shell-setup/` - Modular shell installer (package groups)

Why this structure?
• Clean separation between interface (bin) and logic (scripts)
• Allows updating implementations without changing PATH
• Modular: each script can be used independently
• Version-controlled configs with atomic symlinks
Setup System Architecture
The setup system is built in layers:

Layer 1: Entry Points
• `install.sh` - Bootstrap on fresh system (installs Git + Bun)
• `interactive-setup.ts` - Main interactive TUI (`bun run interactive`)
• `cli-setup.ts` - Headless/non-interactive (`bun run cli`)
• `cli-manager.ts` - Individual package management (`bun run manager`)

Layer 2: Core Modules
• `packages.ts` - Package definitions (18 categories, 100+ packages)
• `executor.ts` - Command execution with retry logic
• `install-manager.ts` - Advanced installer (retry, rollback, progress)

Layer 3: Configuration
• `dotfiles-config.ts` / `dotfiles-setup.ts` - Symlink management
• `system-config.ts` - GNOME, browser, wallpaper, sudo config
• `scripts-executable.ts` - Make all scripts executable
• `nerd-fonts.ts` - Font installation
• `gh-extensions.ts` - GitHub CLI extensions

Layer 4: Verification
• `verification.ts` - Post-install checks and validation
Error Handling Strategy
Multi-Level Failure Recovery:
1. Auto-retry: Up to 3 attempts with exponential backoff
2. Dependency resolution: Auto-install missing dependencies (e.g., snapd)
3. Graceful degradation: Continue on non-critical failures
4. Rollback: Attempt to remove partially installed packages
5. Progress tracking: Save state to `~/.dotfiles/setup/progress.json`
6. Resume capability: Can continue from last successful step
Package Installation Flow
For each package:
1. Check if already installed (version detection)
2. Determine installation method (APT/Snap/NPM/GitHub/CURL)
3. Install dependencies if needed
4. Execute installation with timeout and error handling
5. Verify installation success
6. On failure: retry with backoff, fallback to alternative method
7. Update progress tracker
8. Run post-install hooks (if defined)
Configuration Management
Symlink Strategy:
• All configs symlinked to dotfiles repo (single source of truth)
• Atomic operations: backup existing → create symlink
• Verification: check symlink targets before and after
• Rollback: restore from backup if symlink fails

Files managed:
• Fish: `~/.config/fish/config.fish` → `dotfiles/cfg`
• Neovim: `~/.config/nvim` → `dotfiles/configs/nvim`
• Kitty: `~/.config/kitty` → `dotfiles/configs/kitty`
• Git: `~/.gitconfig` → `dotfiles/configs/git/.gitconfig`
• WezTerm: `~/.config/wezterm` → `dotfiles/configs/wezterm`
Fish Shell Architecture
Modular organization:
• `core/` - Core setup (colors, env, init)
• `aliases/` - Categorized aliases (dev, git, system, docker, etc.)
• `functions/` - Custom Fish functions

Main entry: `cfg` file sources everything in correct order
Total size: 361 lines of aliases + custom functions
Features: Auto-completion, syntax highlighting, git integration
Performance & Optimization
• Parallel package installations where possible
• Lazy loading of shell functions and aliases
• Cache package availability checks
• Skip already-installed packages (version-aware)
• Minimal shell startup time (Fish config optimized)
• Bun for fast TypeScript execution

Architecture Overview
Scripts are organized in two directories: `bin/` contains executable wrappers on your PATH, while `scripts/` holds the actual implementation logic. Many bin scripts are intelligent wrappers that call their counterparts in scripts/, allowing for modular updates without breaking PATH references.

Script Architecture Patterns
Wrapper Pattern: `bin/script-name` → `scripts/script-name` (implementation)
Direct Executable: Standalone scripts in `bin/` with full logic
TypeScript Projects: Complex tools (docker) built with Bun/Node
Symlinks: Some scripts link to common implementations

All scripts are made executable automatically during setup via `scripts-executable.ts`.

Command Line Arguments
Options
Usage
The setup script supports various command-line arguments for fine-grained control over the installation process.
--dry-run
Preview all installations without making changes. Useful for testing or seeing what would be installed.
--dry-run-section <NAME>
Preview a specific section only (e.g., "essential", "editors", "cli-utils").
--dry-run-interactive
Interactive dry-run mode. Select sections to preview interactively.
Alias: --dry-run-i
--skip-system-update
Skip the apt update/upgrade step. Use when system is already up-to-date.
--skip-fonts
Skip Nerd Fonts installation. Useful if fonts are already installed or not needed.
--verbose, -v
Show detailed installation output. Useful for debugging installation issues.
--quiet, -q
Minimal output, errors only. Reduces verbosity for cleaner terminal output.
--install <NAME>
Install a specific section non-interactively. Pass the category ID (e.g., "essential", "editors").
--install-interactive
Select sections to install interactively. Opens category selection menu.
Alias: --install-i
--skip <NAME>
Skip a specific section during installation. Pass the category ID to skip.
--skip-interactive
Select sections to skip interactively. Opens category selection menu.
Alias: --skip-i
--list
List all available packages by category. Exits after displaying the list.
--all, --yes, -y
Non-interactive mode. Installs all categories without prompts. Useful for automation.
Examples:
bun run setup --dry-run
bun run setup --all --skip-fonts
bun run setup --install essential --verbose
bun run setup --dry-run-section cli-utils
